# Mac 平台轻量化 Markdown 阅读器总体方案

## 1. 项目目标

设计并实现一款在 macOS 平台可用的轻量化 Markdown（`.md`）文件阅读器，重点满足以下目标：

1. 打开快、占用小、界面简洁。
2. 以“阅读”为核心，而不是重度编辑器。
3. 支持本地 Markdown 文件与常见资源引用。
4. 对中文内容、代码块、表格、图片有较好的展示效果。
5. 后续可逐步扩展，但首版应尽量简单稳定。

---

## 2. 产品定位

这款应用不是要和 VS Code、Typora 这类全功能编辑器正面竞争，而是定位为：

**一个专注于本地 Markdown 文件浏览与沉浸阅读的轻量桌面应用。**

适用场景：

- 阅读知识库文档
- 浏览项目说明文档 `README.md`
- 查看个人笔记
- 快速预览 Markdown 输出效果

不作为首版重点的能力：

- 协同编辑
- 富文本编辑
- 插件系统
- 云同步
- 复杂笔记管理

---

## 3. 核心功能范围

### 3.1 首版必须具备（MVP）

1. **打开单个 Markdown 文件**
   - 支持 Finder 双击关联打开
   - 支持应用内“打开文件”
   - 支持拖拽 `.md` 文件到窗口

2. **Markdown 渲染阅读**
   - 标题、段落、引用、列表
   - 代码块、行内代码
   - 表格
   - 图片
   - 分隔线
   - 链接

3. **基础阅读体验**
   - 滚动阅读
   - 目录导航（根据标题自动生成 TOC）
   - 页面内锚点跳转
   - 搜索当前文档关键字
   - 字体大小调整
   - 浅色 / 深色主题

4. **资源处理**
   - 支持相对路径图片加载
   - 支持本地文件中的相对链接解析

5. **最近打开记录**
   - 最近文件列表
   - 启动后快速重新打开

### 3.2 第二阶段可扩展

1. 文件夹侧边栏浏览
2. 多标签页阅读
3. 导出 PDF
4. 自动监听文件变化并刷新
5. 阅读位置记忆
6. 自定义 CSS 主题

### 3.3 暂不建议首版实现

1. Markdown 编辑器
2. 所见即所得编辑
3. 云端同步
4. 插件市场
5. 全文数据库索引

---

## 4. 用户体验设计

### 4.1 界面结构

建议采用三栏或两栏的轻量布局：

1. **顶部工具栏**
   - 打开文件
   - 返回/前进（可选）
   - 搜索
   - 主题切换
   - 字号缩放

2. **左侧目录栏**
   - 展示文档标题树
   - 点击跳转到对应章节

3. **右侧主阅读区**
   - 渲染后的 Markdown 内容
   - 流式滚动阅读

首版如果想更轻，可以先做“两栏”：

- 左侧：目录
- 右侧：阅读区

### 4.2 视觉风格

目标是“安静、克制、偏原生”：

- 使用 macOS 风格窗口和控件
- 默认留白充足
- 中文字体显示清晰
- 代码块使用轻量高亮主题
- 避免复杂装饰和厚重配置项

### 4.3 交互原则

- 打开即读，减少设置
- 常用操作一屏可见
- 渲染稳定优先于炫技
- 失败场景给出明确提示，例如图片找不到、链接无法访问

## 5. 技术方案总览

## 5.1 推荐架构

采用典型桌面应用分层：

1. **UI 层**
   - 窗口、工具栏、目录栏、阅读区

2. **应用服务层**
   - 文件打开
   - 最近记录
   - 搜索
   - 主题与偏好设置

3. **Markdown 解析与渲染层**
   - 将 `.md` 文本解析为 AST
   - 再渲染为 HTML
   - 在桌面 WebView 中显示

4. **本地资源访问层**
   - 图片路径解析
   - 本地链接跳转
   - 安全访问控制

5. **持久化层**
   - 保存最近文件
   - 保存主题、字体大小、窗口状态

## 6. 实现语言确认：Java

本项目**正式确定使用 Java 作为实现语言**。

选择 Java 的原因不是因为它在所有维度都绝对最强，而是因为它在这个项目里能提供一条很均衡、很稳妥的实现路径：

- 适合桌面应用工程化开发
- 便于保持清晰的分层结构
- Markdown 渲染链路成熟
- 后续扩展到其他平台更方便
- 对 MVP 阶段来说实现风险可控

### 6.1 为什么 Java 能做

Java 在这个项目里需要解决的核心问题有三个：

1. **桌面界面**
2. **Markdown 解析与渲染**
3. **macOS 应用打包**

这三件事 Java 都有成熟方案：

- GUI：`JavaFX`
- Markdown：`commonmark-java` 或 `flexmark-java`
- 打包：`jpackage`

### 6.2 Java 方案的推荐组合

建议使用：

- **JDK 21（LTS）**
- **JavaFX**
- **flexmark-java**
- **WebView（JavaFX 内置）**
- **Maven 或 Gradle**
- **jpackage**

其中最关键的思路是：

**Markdown 不直接手写原生排版控件，而是转成 HTML，在 JavaFX WebView 中展示。**

这样做有几个好处：

1. 开发效率高
2. 样式控制简单
3. 表格、代码块、图片、目录锚点都更容易处理
4. 主题切换可以主要靠 CSS 完成

## 7. 为什么本项目选择 Java

虽然市场上还有别的可选技术路线，但本项目已经确定使用 Java，因此这里不再做横向摇摆式比较，而是说明为什么 **Java 适合这个项目**。

### 7.1 Java 的优点

1. 工程体系成熟
2. 代码结构清晰，适合长期维护
3. JavaFX 足够完成轻量桌面阅读器
4. `WebView + HTML` 路线实现难度较低
5. 后续如果要支持 Windows / Linux，迁移成本较小

### 7.2 需要接受的现实约束

1. macOS 原生感不如 SwiftUI 强
2. JavaFX 生态不如 Web 前端活跃
3. 打包体积通常不会是最小
4. 如果只做 Mac 独占应用，Java 并不是最贴合平台特性的选择

### 7.3 本项目的选择结论

本项目后续的设计、编码、打包与发布都以 **Java 技术栈** 为基础展开，不再切换到其他实现语言。

一句话总结：

**Java 是本项目的确定实现语言，目标是用稳妥、清晰、可维护的方式完成一款轻量化 macOS Markdown 阅读器。**

## 8. 推荐实现路线

## 8.1 首选路线：JavaFX + WebView

### 核心思路

流程如下：

`Markdown 文件 -> 解析为 HTML -> 注入样式 -> WebView 展示`

### 适合原因

1. Markdown 本质上更适合渲染成 HTML
2. WebView 更容易支持代码块、表格、图片、锚点
3. 主题切换只需要切换 CSS
4. 搜索、滚动定位、标题跳转都比较自然

### 主要模块

1. `FileManager`
   - 打开文件
   - 读取文本
   - 路径规范化

2. `MarkdownService`
   - Markdown 解析
   - HTML 渲染
   - 目录提取

3. `ResourceResolver`
   - 处理相对图片路径
   - 处理本地超链接

4. `ReaderController`
   - 协调 UI 和业务逻辑

5. `PreferenceService`
   - 保存主题、字号、最近文件

6. `SearchService`
   - 当前文档内关键词搜索与定位

## 9. 模块设计

### 9.1 文件打开模块

职责：

- 选择文件
- 校验扩展名
- 读取文件内容
- 记录最近文件

输入：

- 文件路径

输出：

- Markdown 文本
- 文件元信息

异常处理：

- 文件不存在
- 无读取权限
- 编码异常

建议默认支持 UTF-8，并对 BOM 做兼容。

### 9.2 Markdown 渲染模块

职责：

- 解析 Markdown
- 生成 HTML
- 生成目录树
- 为标题生成锚点 ID

建议重点支持：

- GitHub Flavored Markdown 常见能力
- 表格
- 围栏代码块
- 自动链接

### 9.3 阅读展示模块

职责：

- 在 WebView 中显示 HTML
- 应用主题 CSS
- 控制滚动、缩放、跳转

建议：

- 阅读区只负责展示，不承担复杂编辑逻辑
- 搜索定位可以通过注入少量 JS 实现

### 9.4 目录导航模块

职责：

- 从标题提取层级结构
- 左侧展示树状目录
- 点击后滚动到对应锚点

### 9.5 偏好设置模块

保存内容：

- 最近文件列表
- 主题模式
- 字体大小
- 窗口大小与位置

存储方式：

- Java Preferences API
  或
- 本地 JSON 配置文件

首版更建议用 **JSON 配置文件**，可读性更好，也方便以后迁移。

## 10. 技术选型建议

### 10.1 UI 框架

**推荐：JavaFX**

理由：

- 比 Swing 更现代
- 自带 WebView
- CSS 样式能力更强
- 更适合做轻量桌面工具

不推荐首版使用 Swing，除非团队对 Swing 非常熟。

### 10.2 Markdown 解析库

推荐优先级：

1. **flexmark-java**
2. `commonmark-java`

建议首选 **flexmark-java**，因为它扩展能力更强，对表格、目录、更多 Markdown 变体支持更好。

### 10.3 高亮方案

可选方式：

1. 服务端渲染 HTML 时加入代码块 class
2. 在 WebView 中配合 `highlight.js` 做前端高亮

首版建议：

**使用 `highlight.js` + 本地静态资源**

原因是实现简单、效果稳定、主题切换方便。

### 10.4 打包方案

**推荐：jpackage**

输出：

- `.app`
- `.dmg`（可选）

这样可以让应用在 macOS 上以常规方式安装和运行。

## 11. 性能与轻量化策略

既然目标是“轻量化”，设计上要主动克制：

1. **单窗口优先**
   - 首版不急着做复杂多标签体系

2. **本地渲染优先**
   - 不引入远程服务

3. **避免大而全编辑功能**
   - 聚焦阅读体验

4. **按需加载资源**
   - 大图片延迟显示或限制最大宽度

5. **缓存最近渲染结果**
   - 同一文件重复打开时减少重复计算

6. **控制依赖数量**
   - 只保留 UI、Markdown、打包核心依赖

7. **CSS/JS 静态资源本地化**
   - 不依赖网络

## 12. 兼容性与风险点

### 12.1 可能的技术风险

1. **本地图片路径兼容**
   - 相对路径与空格路径处理要仔细

2. **WebView 与本地文件安全策略**
   - 需要设计资源加载方式

3. **不同 Markdown 方言差异**
   - 首版要明确支持范围

4. **超大文件性能**
   - 几 MB 以上 Markdown 渲染要做测试

5. **macOS 打包与签名**
   - 内部分发可以先不做签名
   - 若面向公众分发，要考虑 Apple 开发者签名与 notarization

### 12.2 规避策略

1. 先支持标准 Markdown + 常见 GFM 特性
2. 图片和链接统一做绝对路径解析
3. 对超大文件设置基础保护策略
4. MVP 阶段先以内部分发为目标
